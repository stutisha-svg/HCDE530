# Week 6 — Process

This week I built two Plotly visualizations on a corpus of AI-themed Bluesky posts and ended up needing to enrich the dataset mid-stream to make the second one work. The first chart is a radial VADER sentiment scatter — each post longer than 50 words sits on a circle whose radius is the strength of its sentiment (neutral in the middle, opinionated fanning outward), colored red/grey/blue, with a theme dropdown filtering by topics like `agi_risk` or `ai_creative`. It took several iterations to get readable: filter out short noise, scale opacity to intensity so the impassioned posts pop and the lukewarm ones fade, wrap hover text, move the dropdown to the bottom-left so it stops blocking the title, and box the sentiment legend so it visually reads as a toggle. The second chart, a Sankey tracing one viral post into Reposts / Replies / Quotes and then into composite quality buckets (substance × sentiment), looked sparse because the original query-based collector rarely captured a viral post's actual descendants — eight traceable replies and quotes is not a story. To fix that I wrote `enrich_bsky_threads.py`, which calls Bluesky's `getPostThread` and `getQuotes` endpoints for the top 20 viral root posts and appends roughly 4,000 new reply / quote rows to the CSV. The enrichment immediately exposed two downstream bugs — `NaN` text from thread stubs crashing VADER, and AT-protocol URIs polluting the radial chart's theme dropdown — both fixed with a `.fillna("")` before sentiment scoring and a filter that excludes any 'theme' starting with `_` or `at://`. The whole process turned out to be a useful loop: collect, visualize, notice where the data was too thin to support the chart, write a targeted API enrichment, and patch the visualizations to handle the richer (and messier) data.
