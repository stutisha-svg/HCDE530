# Week 4 Reflection: APIs, Data Acquisition, and Code Literacy

This week I practiced two core competencies: C4 (APIs and data acquisition) and C2 (code literacy and documentation). I chose the PokéAPI because each Pokemon entry includes many structured fields (often 20-40+), which made it a good dataset for practicing endpoint selection, parsing, and filtering logic.

For C4, I started by reading the API docs and narrowing scope to fields that were useful for analysis. I focused on `name`, `types`, `base_experience`, `height`, `weight`, and the six core battle stats (`hp`, `attack`, `defense`, `special_attack`, `special_defense`, `speed`). My analysis angle was to compare stat profiles across overlapping type contexts (for example, dark/ghost and mixed dual types). I intentionally skipped very high-volume or low-signal fields such as moves (50+ per Pokemon in many cases), sprite URLs, and game index lists because those would make a CSV unwieldy without improving my specific question.

The script structure reflects data acquisition best practices. I used helper functions (for example `collect_candidate_payloads`, `normalize_types`, and `fetch_rows`) so that endpoint calls, filtering logic, and CSV writing are separated and testable. I filtered to Platinum-era species with `if sid and sid <= GEN4_MAX_ID`, and enforced a minimum dataset size in `fetch_rows` to ensure I exported at least 50 rows. I also added small `time.sleep()` delays between requests as good API citizenship to avoid hammering the service.

For C2, I focused on writing code that is readable by someone other than me. I kept `main()` short and used descriptive helper names so the control flow is easy to scan. I added inline comments to explain what each endpoint returns, what path parameters are used, and what each parsed key means in the CSV output. I also documented interpretation details that can cause mistakes, especially units (for example, Pokemon height is in decimeters and weight is in hectograms, so a raw value like `7` is not 7 meters). This made the script easier to modify and explain.

I also learned that clean structured data from an API can still be misleading if not validated. Before presenting findings from any large API pull (for example, 2,400 reviews), I would verify: duplicates, bot/synthetic entries, uneven distribution across ratings/stats, and temporal bias (for example, data skewed to recent periods). The API can return valid JSON while still reflecting incomplete, biased, or noisy source data.

Overall, this week helped me connect documentation reading, safe acquisition practices, and code clarity into one workflow: read docs, choose meaningful fields, parse carefully, document assumptions, and validate data quality before analysis.
